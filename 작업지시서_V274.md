# 작업지시서 V274: cai-print 투명 브릿지(Invisible Iframe Bridge) 연동 전략

사용자 답변(Q1: 제어 가능, Q2: 브라우저 내부 통신, Q3: 완전 은닉)을 반영하여, `cai-print`를 `change-viewpoint`의 보이지 않는 하위 서비스로 통합하는 **'Invisible Iframe Bridge'** 전략을 제안합니다.

---

## 1. 아키텍처 설계 (Architecture Design)

### [Main Concept: Invisible Agent Interaction]
- **통신 주체**: `change-viewpoint` (부모) <--> `cai-print` (자식 Iframe)
- **가시성**: 0x0 픽셀의 투명한 `iframe`을 사용하여 사용자 눈에는 전혀 노출되지 않습니다.
- **연동 방식**: `window.postMessage` API를 이용한 양방향 비동기 통신.

## 2. 세부 작동 프로세스 (Workflow)

1. **브릿지 활성화 (Bridge Init)**:
   - `change-viewpoint`에서 'PRINT' 실행 시, `cai-print` URL을 담은 숨겨진 `iframe`을 캔버스 외곽에 동적으로 삽입합니다.
2. **명령 전달 (Command Dispatch)**:
   - 부모 창에서 `iframe.contentWindow.postMessage({ type: 'PRINT_START', payload: {...} })`를 발송합니다.
   - 이때 `payload`에는 현재 캔버스의 AEPL 파라미터와 프롬프트가 포함됩니다.
3. **무인 생성 (Silent Generation)**:
   - `cai-print`는 메시지를 수신하면 화면 렌더링 없이 즉시 '인쇄 모드'로 진입하여 결과 이미지를 생성합니다.
4. **이미지 수집 (Data Capture)**:
   - 생성 완료 후, `cai-print`는 `window.parent.postMessage({ type: 'PRINT_COMPLETE', data: 'base64...' })`를 통해 결과물을 부모 창으로 반환합니다.
5. **결과 반영**:
   - `change-viewpoint`는 이미지를 수신하여 로컬에 저장하거나 캔버스에 즉시 렌더링한 뒤, 사용했던 `iframe`을 제거(Cleanup)합니다.

---

## 3. 제안 및 확인 사항 (Propositions & Questions)

본 전략의 성공적인 구현을 위해 다음 3가지 방안 중 하나를 선택해 주세요.

- **방안 A (URL 통합형)**: `cai-print`가 URL 파라미터(`?mode=hidden&prompt=...`)를 지원하도록 수정하여, 로드 즉시 생성이 시작되도록 함. (가장 간단)
- **방안 B (Event 통신형)**: Iframe 로드 후 `postMessage`로 정교하게 상태를 제어함. (확장성이 높음)
- **방안 C (Background API Polling)**: Iframe은 단순히 생성 신호만 보내고, `change-viewpoint`가 로컬 서버의 `cai-print` 폴더를 직접 감시하여 결과를 가져옴. (안정성이 높음)

---

> [!TIP]
> **권장 사항 (Best Practice)**: `방안 B (Event 통신형)`를 추천합니다. 진행 중인 상태(진행률 등)를 부모 창의 `analysisStep`에 실시간으로 표시할 수 있어 사용자 경험이 가장 뛰어납니다. 확인 후 승인해 주시면 구현을 시작합니다.

# 작업지시서 V273: cai-print 연동 및 이미지 캡처 전략 수립

본 문서는 `cai-print` 서비스를 새 탭으로 연동하되, 사용자에게는 해당 탭을 노출하지 않고(Hidden) 최종 생성된 이미지만을 `change-viewpoint`로 가져오는 최적의 전략을 제안합니다.

## 1. 해결 과제 (Challenges)
- **탭 숨김 (Hidden Tab)**: 최신 브라우저의 보안 정책상 완전히 보이지 않는 새 탭은 불가능하므로, 사용자 경험(UX) 측면에서 '보이지 않게 처리'하는 기술적 우회로가 필요합니다.
- **이미지 수집 (Image Capture)**: 다른 도메인 또는 탭에서 생성된 이미지를 보안(`CORS` 등) 이슈 없이 부모 창인 `change-viewpoint`로 안전하게 전달해야 합니다.

---

## 2. 제안 전략 (Proposed Strategies)

### [Option 1] Invisible Iframe + postMessage (권장)
- **방식**: `change-viewpoint` 내부에 보이지 않는 `iframe`(0x0 크기 또는 `display: none`)을 생성하여 `cai-print`를 로드합니다.
- **작동**: URL 파라미터로 명령을 전달하고, `cai-print`에서 생성 완료 시 `window.parent.postMessage()`를 통해 Base64 이미지 데이터를 전송합니다.
- **장점**: 사용자에게 전혀 노출되지 않으며, 실시간으로 데이터를 주고받을 수 있어 통합감이 가장 뛰어납니다.

### [Option 2] Background Popup + Local Storage (분리형)
- **방식**: `window.open`을 사용하여 `cai-print`를 팝업으로 띄운 뒤, `window.blur()` 또는 `focus()` 제어를 통해 백그라운드에서 실행되도록 유도합니다.
- **작동**: `cai-print`가 생성된 이미지를 저장(Local API)하거나 `window.opener`로 전달합니다.
- **장점**: `iframe` 제약(일부 사이트의 임베딩 거부)이 있는 경우에도 안정적으로 작동합니다.

### [Option 3] API-based Capture (Direct Bridge)
- **방식**: 탭을 띄우지 않고, `cai-print`의 백엔드 엔드포인트에 직접 요청을 보내 결과물을 JSON/Base64로 받습니다.
- **작동**: `change-viewpoint` -> `cai-print` API -> 이미지 생성 -> 결과 반환.
- **장점**: UI 의존성이 전혀 없어 가장 견고하지만, `cai-print` 쪽에 API 인터페이스가 구축되어 있어야 합니다.

## 3. 사용자 확인 및 질문 (Questions)

전략 수립을 위해 다음 사항에 대한 확인이 필요합니다.

1. **`cai-print` 서비스의 제어권**: `cai-print` 앱의 소스코드를 수정하여 `postMessage`와 같은 통신 코드를 추가할 수 있는 상태인가요?
2. **데이터 수신 방식의 선호도**: 브라우저 내부 통신(Iframe/Message)을 선호하시나요, 아니면 로컬 물리 폴더(/api/save-images)에 파일이 생성되는 것을 감시하여 가져오는 방식을 선호하시나요?
3. **'Hidden'의 정의**: 사용자 눈에 전혀 보이지 않아야 하나요(Iframe), 아니면 백그라운드 탭으로 열려도 무방한가요?

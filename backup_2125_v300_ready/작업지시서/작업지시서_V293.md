## Q&A 확정

### BUG-1 (Change A) 근거
`layerC_property`가 Phase 4 생성 프롬프트에 전달하는 BIM 파라미터가 구버전 키(`1_macro_geometry`, `3_material`, `4_fenestration`)를 사용하여 모두 `'N/A'` 출력.
현행 AEPL v4 스키마 키: `1_Geometry_MASTER`, `2_Property_SLAVE`.
→ 입면도의 재료·형태 정보가 Phase 4에 전달되지 않음 = 입면도-생성 이미지 불일치의 직접 원인.

### BUG-2 (Change B) 근거
ANGLES 상수: `['12:00', '1:30', '3:00', '04:30', ...]` (일부 leading zero 없음)
`getSemanticAngle` switch 케이스: `case '03:00'`, `case '01:30'` (leading zero 있음)
→ `'1:30'`(우후방), `'3:00'`(우측) 선택 시 시맨틱 설명 누락 → Blind spot 시점 품질 저하 직접 원인.

### Phase 0 목적 확인 (Change D)
이미지 재생성(Phase 0)의 목적: AI가 원본 이미지를 완전히 소화(내재화)한 뒤 분석 수행.
→ **방안 1 채택**: Phase 0 이미지 재생성 → 텍스트 묘사 생성(이미지 미생성)으로 교체.
- 원본 이미지 불변 → BUG-3(기하학 불일치) 해결
- API 호출 횟수 현행 유지 (2회), 이미지 생성 비용 → 텍스트 응답으로 절감
- AI가 생성한 묘사를 Phase 1 분석 컨텍스트로 주입 → 소화 효과 동일

### Change A — AEPL 키 통일

**확정 효과**

현행 `layerC_property`의 4개 필드(`Mass Typology`, `Roof Form`, `Base Material`, `Fenestration`)가 모두 `'N/A'`로 Phase 4 프롬프트에 전달됨. 수정 후 Phase 1에서 추출한 실제 AEPL v4 데이터가 정확히 전달됨.

```
수정 전: Step 5 프롬프트 = "Mass Typology: N/A / Roof Form: N/A / ..."
수정 후: Step 5 프롬프트 = "Mass Typology: Podium / Roof Form: Flat / Base Material: Concrete / ..."
```

**불명확 요소**

| # | 불명확 사항 | 이유 |
|---|------------|------|
| A-1 | BIM 텍스트 데이터가 IMAGE_GEN 모델의 이미지 생성 품질에 실제로 영향을 주는가 | AEPL 파라미터는 텍스트 프롬프트로 주입되나, IMAGE_GEN 모델이 텍스트 구조 지시를 얼마나 반영하는지는 모델 특성에 따라 다름 |
| A-2 | 입면도 이미지(Phase 2)와 BIM 파라미터(Phase 4 텍스트)가 충돌할 경우 어느 쪽이 우선되는가 | Phase 4는 원본 이미지 + 입면도 이미지 + 텍스트 파라미터 3가지를 동시에 참조함 |

**최적 제안**

- A-1: 수정 전후 동일 건물·동일 시점으로 생성 비교 테스트 → `layerC_property` 블록이 출력 이미지에 미치는 영향 실측 확인
- A-2: 현재 프롬프트 구조(`Geometric Sanctuary: The building's proportions are Immutable Constants`)는 이미지 참조를 우선시 → 텍스트는 보조 역할이므로 충돌 위험 낮음, 현행 구조 유지

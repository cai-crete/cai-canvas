**작성일**: 2026-03-31

## 1. 프로토콜 분석 및 오염 요소(Deviations) 파악

"System Protocol Image to Elevation-v2.2" 및 "Macro-AEPS-V2" 원본 문서를 바탕으로 현재 `App.tsx`의 생성 로직(`generateElevations`)을 진단한 결과, 시스템 근본 철학을 위반하는 **3가지 치명적 오염**이 발견되었습니다.

### ❌ 오염 1: 단방향 데이터 흐름(DAG) 위반 — V300 스택링 엔진 설계의 오류
*   **프로토콜 명시**: 시스템 아키텍처는 "단방향 게이트 제어형 상태 기계(DAG)"이며, "정보는 폭포수처럼 하위 단계로만 흐름(Unidirectional Lock-on)"을 원칙으로 합니다. 독립된 5개 뷰포인트 전개도를 "동시 출력"해야 합니다.
*   **현재 구현 (V300)**: FRONT를 먼저 생성하고, 그 결과물(출력 데이터)을 다시 입력 데이터로 재귀 삽입(Stacking)하여 나머지 4개 뷰를 생성하고 있습니다. 이는 출력물이 입력으로 **역류(Backflow)**하는 구조이며, Protocol의 폭포수 모델을 파괴하는 편법입니다.

### ❌ 오염 2: 모달리티 분리 원칙(ensemble_pair) 위반 — 텍스트/이미지 프롬프트 혼용
*   **프로토콜 명시**: `1_Geometry_MASTER`는 오직 형태 확정자(Image Prompt)로, `2_Property_SLAVE`는 오직 정보 귀속자(Text Prompt)로 분리되어야 합니다. 또한 Protocol B Phase 2는 "형태 제어 단어의 완벽한 소거(Anti-Morphology Filter)"를 강제합니다.
*   **현재 구현**: API 호출 시 텍스트 프롬프트 안에 `contextualParamsStr`를 통째로 주입하고 있습니다. 이 안에는 `1_Geometry_MASTER`의 기하학 데이터까지 텍스트로 섞여 있어, 텍스트가 형태에 간섭(미세 환각, Greebling)하는 원인을 제공합니다. 

### ❌ 오염 3: 3D 벡터와 2D 이미지의 물리적 충돌 — TOP 뷰 실패의 근본 원인
*   **프로토콜 명시**: 각 뷰는 절대적인 법선 벡터(Normal Vector)를 갖습니다. TOP 뷰는 `(0, 0, 1)`, 즉 완벽한 수직 하향 시점이어야 합니다.
*   **현재 구현**: 평면 벡터 `(0, -1, 0)`를 지닌 정면(FRONT) 이미지를 모든 뷰의 "형태 앵커"로 강제 삽입했습니다. 3D 공간 상에서 정면 이미지는 Z축(지붕) 데이터를 전혀 갖고 있지 않는데, 이를 바탕으로 Z축 수직 뷰(TOP)를 그리라고 명령하므로 물리적 모순이 발생, AI가 명령을 붕괴시키고 정면을 재출력하게 됩니다.

## 2. 수행 계획 (V302 복원 솔루션)

ELEVATION 생성 로직을 Protocol B와 Macro-AEPS-V2의 순수 결정론적 파이프라인으로 100% 롤백 및 재구축합니다.

### Change A. 역류 차단 및 5면 병렬 파이프라인 복원 (V300 페기)
순차적 스택링(FRONT → OTHERS) 코드를 전면 폐기하고, 오직 원본 이미지(Source Image) 1장만을 바탕으로 5개의 뷰를 완전 병렬(Promise.all) 로 일괄 렌더링하도록 롤백합니다. 

### Change B. `ensemble_pair` 모달리티 완벽 분리 적용
`App.tsx` 내에서 AEPL 파라미터를 파싱하여 형태와 속성을 강제로 분리합니다.
1.  **Image Prompt (`inlineData`)**: 원본 소스 이미지 1장만 삽입 (Protocol B Phase 1의 3D Whitebox 추출 기반 시각 앵커용).
2.  **Text Prompt (`text`)**: `extractedParams` 배열에서 `1_Geometry` 관련 블록을 삭제하고, **오직 `2_Property` (재질, 색상, 로이유리, 빛 등) 객체만 필터링하여 텍스트로 주입**합니다. 프롬프트 내에 형태를 제어하는 명사가 침투하는 것을 차단합니다.

### Change C. Protocol B 렌더링 프롬프트 재각인 (Internal 3D Engine 명령)
AI 애플리케이션 한계상 실제 3D 엔진(WebGL 등)이 내장되어 있지 않으므로, Gemini Vision 모델 자체가 내부적으로 3D Whitebox 변환을 온전히 수행하도록 프롬프트를 프로토콜에 맞게 재작성합니다.

```text
[PROTOCOL B: Visualization Execution Engine v2.2]
TASK: Deterministic Projection of the building's ${config.label} Orthographic View.

[PHASE 1: GEOMETRY MASTER LOCKING (Image Prompt Execution)]
1. Analyze the attached SOURCE IMAGE. Internally construct a rigid 3D Whitebox of its bounding volume.
2. Re-project this 3D Whitebox strictly constrained to the Absolute Camera Vector: 
   - Azimuth: ${config.az}°
   - Altitude: ${config.alt}°
   - Normal Vector: ${config.normal}
CRITICAL FOR TOP VIEW (Alt 90): You MUST look strictly down upon the roof. Do NOT project the front facade.

[PHASE 2: PROPERTY SLAVE INJECTION (Text Prompt Execution)]
- Texture & Material Guidelines:
${onlyPropertyParamsStr} -> Cast these surface properties onto the re-projected 3D geometry.
- Constraint: Apply materials ONLY. Do not alter the Phase 1 geometric projection.

CONSTRAINTS: NO text, pure transparent background.
```

## 3. 제안 사항 (Q1)

Change B(모달리티 분리) 구현 방식에 대한 질문입니다. 현재 `extractedParams`로 넘어오는 JSON 포맷을 분해하여 텍스트 프롬프트에 넣을 데이터를 선택해야 합니다.

*   **제안 1 (수동 필터링 - 권장)**: `App.tsx` 레벨에서 `extractedParams` 객체의 키를 검사하여, 이름에 `1_Geometry` 또는 `macro`가 포함된 키는 버리고, `2_Property` 또는 `material`이 포함된 키값만 `JSON.stringify` 하여 프롬프트(Phase 2)에 꽂아 넣습니다. (프로토콜의 Anti-Morphology 완벽 준수)
*   **제안 2 (전체 주입 후 프롬프트단 차단)**: 기존처럼 전체 JSON을 텍스트로 넣되, 시스템 프롬프트에 "JSON 중 1_Geometry 파트는 읽기만 하고 형태 변경에 쓰지 마라"고 지시합니다. (구현은 쉽지만 미세 환각 확률 잔존)

답변해 주시면 V300을 폐기하고 V302로 완벽한 프로토콜 정합성 코드를 구현하겠습니다!

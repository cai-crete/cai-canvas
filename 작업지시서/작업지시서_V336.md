# 작업지시서 V336: Integration Process 아키텍처 재정의 (프론트엔드 역할 명세)

## 1. 작업 개요
본 작업은 현재 앱(`cai-canvas`)의 "Integration Process"를 전면 재수정하는 문서화 작업입니다. 
가장 중요한 핵심은 **"하네스 엔지니어링(Harness Engineering) 구축은 본 앱 내부가 아닌 외부(별도 시스템)에서 이루어진다는 점"**입니다. 따라서 본 문서 업데이트의 목적은, 외부 하네스 시스템이 원활하게 연동될 수 있도록 현재 앱의 역할을 **'순수 무한 캔버스 프론트엔드'**로 명확히 규정하고 그 상태를 정리하는 것입니다.

## 2. 권한 및 역할의 분리 (Architecture Redefinition)

### 2.1. 외부 하네스 엔지니어링 (External Harness System - 미래 구축 대상)
- **역할**: 업로드된 사진 데이터를 분석하고, 캔버스의 좌표/맥락을 유지하며, 각 백엔드 모듈(CHANGE VIEWPOINT, PLANNERS 등)로의 트래픽 라우팅 및 연산 오케스트레이션을 담당하는 핵심 미들웨어/컨트롤러.
- 본 앱(`cai-canvas`)은 이러한 연산 로직을 직접 수행하지 않고, 외부 하네스로 위임하는 구조를 지향합니다.

### 2.2. 무한 캔버스 프론트엔드 (Current App: `cai-canvas`)
- **역할**: 시각적 I/O (Input/Output) 및 사용자 인터페이스(UI) 제어.
- **주요 기능**:
  - **이미지 주입(Ingestion)**: 사용자가 프론트엔드인 캔버스에 사진을 업로드하고 배치함.
  - **Payload 준비**: 업로드된 이미지 및 캔버스 상의 뷰포트 상태를 기초 데이터로 묶어(Context) 외부 하네스/백엔드로 전송할 준비를 함.
  - **결과 출력(Rendering)**: 외부 시스템을 거쳐 백엔드에서 생성된 JSON 파라미터 및 이미지 데이터를 수신하여, 캔버스 위에 정확한 위치와 상태로 노출함.

### 2.3. 백엔드 연동 구조 (Decoupled Backend Services)
- `IMAGE TO ELEVATION`, `CHANGE VIEWPOINT`, `SKETCH TO IMAGE` 등은 각기 분리된 프롬프트와 분석 로직을 가진 독립된 백엔드 서비스망(Server API)으로 간주됩니다.
- 기존처럼 앱 내부에서 이들 간의 파라미터 종속성(의존성)을 강제하지 않습니다.

## 3. 세부 문서 업데이트 계획 (`Protocol/Integration Process.md`)

- **문서 서문 변경**: 기존의 복잡한 5-IVSP 등 모놀리식 추론 로직 설명을 축소하고, 본 앱의 목적이 "외부 하네스 엔지니어링과 백엔드 연동을 위한 무한 캔버스 프론트엔드 I/O 제어기"임을 선언.
- **프로세스 재편**:
    - **PHASE 1: I/O Ingestion (프론트엔드 입력)** - 사진 업로드 및 캔버스 좌표/UI 상태 초기 설정.
    - **PHASE 2: Harness & Backend Routing (외부 위임)** - 선택한 패널 툴에 따라 분석(ANALYZE) 요청을 외부로 전송.
    - **PHASE 3: Canvas Synchronization (프론트엔드 출력)** - 수신된 결과 데이터와 이미지를 캔버스 스토어에 동기화하고 시각화.
- **Legacy 종속성 삭제**: 과거 `IMAGE TO ELEVATION`이 다른 모든 작업의 강제 선행 조건이었던 낡은 설명을 제거.

## 4. 진행 절차
1. 본 작업지시서(V336) 확인 및 승인.
2. `Protocol/Integration Process.md` 문서 내용을 위 구조에 맞게 전면 리팩토링.
3. 문서 내 프론트엔드 관점의 I/O 통신 흐름 다이어그램/리스트 정리.
4. 작업 완료 후 리뷰 및 V336 마무리.

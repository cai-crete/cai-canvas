# 작업지시서 V319 (CANCEL 버튼 클릭 시 API 전송 중단 처리)

**작성일**: 2026-04-01
**버전**: V319
**상태**: 검토 대기

---

## 1. 개요 및 목표
"CANCEL" 버튼 클릭 시 UI 상태만 초기화하는 것이 아니라, 실제 진행 중인 비동기 API 요청(Gemini SDK / fetch)을 즉시 중단(Abort)하여 불필요한 리소스 소모 및 할당량(Quota) 낭비를 방지합니다.

## 2. 주요 분석 내용
*   **현재 상태**: `handleCancelGenerate` 함수가 `abortControllerRef.current.abort()`를 호출하고 있으나, 실제 `handleAnalyze`, `handleGenerate` 등 비동기 함수들이 이 신호(Signal)를 API 호출부에 전달하지 않거나, 중간 단계에서 체크하지 않아 프로세스가 멈추지 않음.
*   **해결 방안**:
    1.  **AbortController 초기화 및 전달**: 모든 비동기 진입점(`handleAnalyze`, `handleGenerate`) 시작 시 `AbortController`를 초기화합니다.
    2.  **API 호출 시 Signal 주입**: `fetch` 및 지원 가능한 경우 Gemini SDK 호출부에 `signal: controller.signal`을 주입합니다.
    3.  **단계별 중단점(Guard Clause) 배치**: `await` 직후마다 `signal.aborted` 여부를 확인하여 프로세스를 조기에 종료합니다.

## 3. 주요 업데이트 계획

### ✅ 1. AbortController 라이프사이클 통일
*   `handleAnalyze` 및 `handleGenerate` 시작 부분에서 `abortControllerRef.current = new AbortController()`를 반드시 수행합니다.
*   `analyzeViewpoint`, `generateElevations`, `getArchitecturalDescription` 등 하위 함수에 `signal` 객체를 인자로 전달하거나 `ref`를 참조하게 합니다.

### ✅ 2. 비동기 블록 내 중단 로직 강화
*   각 대형 API 호출(`generateContent`) 및 이미지 로드(`img.onload`) 프로세스 직전에 `if (signal.aborted) return;` 체크를 추가합니다.
*   `Promise.all` 내의 병렬 호출부(`generateOne`)에도 중단 신호를 모니터링하도록 수정합니다.

### ✅ 3. 예외 처리 로직 (AbortError)
*   사용자가 의도적으로 취소한 경우(`AbortError`)에는 "실패" 토스트 메시지가 표시되지 않도록 `catch` 블록의 에러 핸들링을 정교화합니다.

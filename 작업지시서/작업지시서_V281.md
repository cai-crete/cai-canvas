# 작업지시서 V281: 시스템 흐름 제어 및 스케치-이미지 동기화 고도화

## 1. 개요 및 요구사항 분석
사용자께서 남기신 7가지 세부 추가 요구사항을 모두 분석하여 기획된 V281 기능 정의서입니다. 파생 뎁스 제어, 레이블 지정의 디테일을 다듬고 특히 객체 리사이징과 펜 스케치(path) 간섭을 정교하게 동기화합니다.

## 2. 세부 기획 및 전략 (Strategy)

### [전략 1] 네이밍 및 상태 정의 룰 확장 (요구사항 1, 7)
- **SKETCH TO IMAGE (`EDIT 01` 기반)**: `EDIT 01` 아트보드에서 스케치 후 생성된 이미지의 레이블을 이전 룰(`EDITED`)에서 정정하여 **`GENERATED 01`(순번 포함)**로 부여합니다. 모체 속성 또한 `EDIT 01` 본인으로 유지되도록 연산합니다.
- **CHANGE VIEWPOINT 모체 명시**: `VIEWPOINT 01` 등 생성된 뷰포인트 이미지의 모체(`motherId`)가 원본 `ANALYZED` 이미지임을 로직 상에서 한 번 더 명확하게 검증 및 고정합니다.

### [전략 2] 이미지 리사이징 시 스케치 선(Path) 동기화 연동 (요구사항 2)
- **개념**: 캔버스 상에서 이미지(`artboard`, `generated` 등)의 크기를 조절 제어점(Handle)으로 당기거나 줄일 때, 해당 이미지 위에 스케치되었던 선들이 제자리에 멈춰있지 않도록 합니다.
- **구현 방식**: `handleResizeMap` 연산 로직 내에서, 변경 대상 이미지의 확대/축소 비율(Scale Ratio X, Y)을 실시간으로 계산하여, 해당 이미지를 부모(`motherId`)로 갖는 모든 `path` 타입 아이템들의 내부 좌표점(`points`) 배열 객체들에 스케일 매핑곱을 수행합니다.

### [전략 3] 모체-생성물 레이아웃 배치 로직 제어 (요구사항 3, 5)
- **초기 아트보드 파생물 (요구사항 3)**: 처음 올린 초기 아트보드(`SKETCH`)로부터의 파생 이미지는 여타 혼선 없이 항상 우측(+120px)으로 뻗어나가도록 (기존 로직 보강) 방어 코드를 작성합니다.
- **VIEWPOINT 편집(복제) 시 독립 분리 (요구사항 5)**: `VIEWPOINT 01` 이미지에서 편집(`[+]` 아이콘)을 클릭할 때, 생성되는 복제 아트보드의 위치를 일반적인 우측이 아닌 **직하단(Y+Height+120)** 으로 배치하며, 동시에 모체 식별자(`motherId`) 값을 모두 끊어내어(`null`) 아예 새로운 파생 역사의 시발점(독립된 모체)이 되게 만듭니다.

### [전략 4] 목적 외 버튼 UI 인터페이스 숨김 처리 (요구사항 4, 6)
- **업로드 버튼 제거 (요구사항 4)**: 파생된 이미지(`GENERATED` 등)로부터 복제되어 뷰가 세팅된 `EDIT 01~` 아트보드의 헤더 제어바에서는, 사용자가 임의의 다른 이미지를 덮어씌우지 못하게끔 **'업로드 구름 아이콘'**을 DOM에서 아예 노출시키지 않습니다.
- **라이브러리(입면 전개) 권한 본체 이전 (요구사항 6)**: 뷰포인트 변환 후 활성화되는 입면 전개도(책 모양 아이콘) 버튼의 렌더링 위치를 생성된 자식들(`VIEWPOINT XX`)의 하위 툴바에서 제거하고, 대신 원본이 되는 **`ANALYZED` (본체)** 이미지의 툴바 중앙에만 표시하도록 UI 권한을 넘깁니다.

## 3. 사용자 확인 및 질문 (Questions)
정확한 의도 파악과 잠재적 모순 방지를 위해 다음 내용을 문의드립니다. 

**Q1. 스케치 선 연동: "드래그 시 이동" 여부**
현재 "이미지 사이즈 조절(Resize) 시 스케치 크기도 함께 변형" 하도록 로직 구축을 준비했습니다. 혹시 이미지 리사이징뿐만 아니라, **이미지를 마우스 클릭 후 이리저리 드래그(+이동)시켜 위치를 바꿀 때에도** 거기에 소속된 스케치 선들이 알아서 같이 묶여 이동해야 하는 구조인가요?

**Q2. 5번 항목 "독립된 모체"의 트리(Tree) 연결성**
VIEWPOINT 영상에서 [+] 를 눌러 나오는 복제 캔버스가 `독립된 모체`라는 의미는, 기존 가계도나 트리에 그려지는 부모-자식 연결선 자체가 완전히 끊어지고 아예 새로운 프로젝트(MotherID: Null)처럼 분리되어 시작된다는 뜻이 맞으신가요?

**Q3. 4번 업로드 버튼의 제거 범위**
제어바 업로드 버튼 제거 대상을 오직 `EDIT 01, EDIT 02...` 이름을 가진 캔버스들에게만 1차원적으로 적용할까요, 아니면 더 확실한 차단을 위해 **"이미 빈 바탕이 아니고 그림(src)이 하나라도 얹혀있는 모든 복제/파생 아트보드"**에는 전체적으로 업로드 아이콘을 제거하는 것이 나을까요?
